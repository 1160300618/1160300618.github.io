BFS算法

概述

Breadth-first-search(BFS)是遍历图的一个一般技术，BFS遍历图G，可以

  • 访问G的所有边和顶点
  • 确定G是否连通
  • 计算G的所有连通分支
  • 计算G的生成森林

BFS遍历具有n个顶点和m条边的图G的时间复杂性为O(n+m)

BFS可以扩展解决其他图论问题

  • 找到两个顶点之间具有最少边数的路径（若所有边权值相同即为两点间最短路径）
  • 找到图中的简单环，如果存在的话

基本思路

白色：未访问顶点 灰色：待访问顶点 黑色：已访问顶点

从顶点v0开始访问

v0已访问，从v0处可以访问v1,v2

v1,v2已访问，从v1处可以访问v0,v2,v3,v5，从v2处可以访问v0,v1,v3, v0,v1,v2已访问，访问v3,v5

从v3,v5 处访问v4,v6

全部访问，终止

java 文件读写

文件读写是Java中的常见操作，流是文件的一个基本概念。文件的读写、网络收发和进程通信都要用到流，通常来说，流有源和目的端，可以是内存区域，也可以是磁盘文件，甚至可以是URL

在java中，每个流都是一个对象，流通常分为两种，输入流和输出流。java.io包中的stream类根据操作对象可以分为两种，字符流和字节流。

字节流：InputStream是所有字节输入流的父类，OutputStream是所有字节输出流的父类。

字符流：Reader是所有字符输入流的父类，Writer是所有字符输出流的父类。

字节：Byte：1 byte = 8 bit
字符：char: 1 char = 2 byte = 16 bit

在ASCII码中，编码了128个字母或标点，将所有编码都完成了，一个字节就可以存储信息。但是对于汉字等文字，一个字节远远不够，所以从字节扩展到字符。

字节流就是普通的二进制流，最小单位是字节，字符流就是在字节流的基础上根据编码方式处理的结果，本质上也是字节流。所以字节流主要操作字节序列的对象，如二进制文件、图片、影像文件等，字符流则主要操作字符序列，如文本呢文件等。

（一）字节流

InputStream和OutputStream是两个抽象类，要进行文件读写的操作，需要使用FileInputStream和FileOutputStream子类。
主要方法：
InputStream:
读取一个字节： public int read() throws IOException
读取一组字节： public int read(byte[]) throws IOException
关闭流： public void close() throws IOException

OutputStream:
写一个字节： public void write(int b) throws IOException
写一组字节： public void write(byte[]) throws IOException
关闭流： public void close() throws IOException

（二）字符流

Reader和Writer是处理字符流的抽象类，使用子类FileReader和FileWriter来创建实例进行读写操作。
Reader:
读取一个字符： public int read() throws IOException
读取一组字符： public int read(char[]) throws IOException
关闭流： public void close() throws IOException

Writer:
写一个字节： public void write(int b) throws IOException
写一组字节： public void write(char[]) throws IOException
写一个字符串： public void write(String string) throws IOException
关闭流： public void close() throws IOException

（三）缓冲流

缓冲流的原理是将数据先缓冲起来，然后一起写入或读出，使用缓冲流非常高效。
常用缓冲流：
BufferedReader, BUfferedWriter
BufferedInputStream, BufferedOutputStream

1、BufferedReader / BufferedWriter

BufferedReader属于字符流，要使用就需要将一个字节流转换成一个字符流，使用InputStreamReader,OutputStreamWriter

BufferedReader,BufferedWriter提供的方法与Reader，Writer类似，可以读单个字符，一组字符，还可以读一行字符，写一个字符，一组字符,一个字符串。

BufferedReader:
读取一个字符： public int read() throws IOException
读取一组字符： public int read(char[]) throws IOException
读一行文件： public String readline() throws IOException
关闭流： public void close() throws IOException

BufferedWriter:
写一个字节： public void write(int b) throws IOException
写一组字节： public void write(char[]) throws IOException
写一个字符串： public void write(String string) throws IOException
关闭流： public void close() throws IOException

2、BufferedInputStream / BufferedOutputStream

提供方法与InputStream,OutputStream相同

（四）Scanner

Scanner类是在java.util包中定义的，主要用来键盘输入数据，也可以文件输入数据，但不能写文件。

Scanner可以用来读入各种数据，int, double等，还可以用has()方法观察是否读完。

Java 多线程

Java线程是程序内部的控制机制，类似于多进程。多进程可以同时进行多个任务，多线程即在一个进程内部多布划分，让多步同时进行。

线程的创建

创建线程有两种方式，继承Thread类或实现Runable接口，每种实现方式都是重写一个自己的run()函数，然后 new 一个线程，但要注意启动线程必须使用start(),不能使用run()。

这是创建线程的方式，注意都是使用start()创建，不能用run()。查看Thread的实现：

run()中target即是默认的执行线程，所以如果要调用run()，会直接执行run()中的代码，重写后即开发者的代码。而start()中首先是检查状态，然后要把这个线程加入到线程组group中，然后更改状态。start0()是一个本地方法，可以看到没有调用run()方法。
Java线程启动是通过start()启动的，但是启动后run()中的代码什么时候执行是由JVM决定的，如果直接调用run()方法，则相当于是一个普通的方法，与线程无关，直接就是一个串行的程序，所以在写代码时注意不要直接调用run()方法。

线程竞争

线程竞争是多个线程在访问同一块内存空间时出现的情况，一个线程内部有多行代码，那么如果两个线程同时更改内存，就可能会出现多种情况。

两个线程都进行add()操作。但add内部实现时，要对内存进行取值、修改、写回。线程2在取值时，线程1可能已经修改过值，也可能未修改，因次可能会出现各种结果，因为线程竞争而出现线程不安全。

线程安全策略

1、限制数据共享

1.将可变数据类型限制在单一线程中，避免竞争,不允许任何线程直接读写该数据
2.线程之间不共享mutable数据类型
3.局部变量一定是线程安全的，局部变量的引用如果指向了外部的对象，要保证引用的对象只在该线程中使用。
4.避免使用全局变量。

这种方式使用范围不大， 核心思想：线程之间不共享mutable数据
不共享数据当然不会出现竞争。

2、共享不可变数据

1.基本数据类型直接加final。
2.可变数据类型必须保证以下几点
（1）没有mutator方法
（2）所有的域都是private和final
（3）没有表示泄漏
（4）没有beneficent mutation
（5）子类型不能重写父类型方法
（6）构造器可以为private，构造时通过工厂方法构造

这种方法和第一种类似，可以共享内存，但内存数据只能read，不能write。将所有的变量都设置为不可变的。

3、使用线程安全的数据类型

synchronizedCollection(Collection c);
synchronizedSet(Set s);
synchronizedList(List list);
synchronizedMap(Map<K,V> m);

都是Java提供的将Set，List这些线程不安全的数据类型转换为线程安全的数据类型的函数。但是
（1）线程安全只是针对单个操作的，例如多个线程都只对list进行add()操作，在内部机制保证下，可以保证线程安全。但若有多个操作组合，还是不安全的。例如

多个线程同时进行时，在第二个线程判断包含之后，第一个线程remove,set已经改变，但第二个线程还要进行remove,因此会造成线程不安全。
（2）用Iterator遍历是不安全的
线程安全的数据类型只是保证数据类型的操作是线程安全的，但不能保证对内存的直接访问是安全的。在遍历时，已经直接访问到对象的内存空间。如果直接对内存进行修改而不通过数据类型的操作进行，那么是不安全的。

4、同步机制共享线程不安全的可变数据

这是保证线程安全的最主要的策略，这种机制是通过对对象的访问进行加锁，使这个进程获得独有的对对象mutation的权利。

在一个线程申请了锁之后，这个锁就是线程独有的，直到synchronized代码块中的代码执行完之前，锁都不会被释放。其他申请锁的线程会被阻塞。当这个线程释放锁，其他线程得到锁之后才能继续进行。

注意：拥有一个对象的锁不是把这个对象上锁不让别的方法访问
synchronized (obj) { … }
in thread t does one thing and one thing only: prevents other threads from entering a synchronized(obj) block, until thread t finishes its synchronized block. That’s it.

锁换句话说只是一个约定，对于遵循这个约定即使用锁的代码块，必须申请到锁才能执行。但是锁可以说与要访问的对象是没有任何关系的，锁不能阻止访问对象。如果线程没有申请对这个对象的锁，那么就不会受到这个约定的限制，那么在执行与对象相关的代码时就不会有申请锁、被阻塞等步骤，而是直接进行对象的操作。

equals()和hashCode()

equals()和hashCode()都是Object类实现的方法，所以在所有的类中都有。equals（）用于比较两个对象是否等价，hashCode()是一个从对象到散列值的映射，以支持Java中基于hash的HashSet, HashMap等数据结构。

Object类实现的equals()是用 “==”实现的，所以在ADT中必须重写equals()和hashCode()。

（一） == 和 equals()

1、== ：引用等价性，比较两个对象的引用是否相同，即两个对象是否指向同一个内存空间。

2、equals()：对象等价性，按照自定义的规则比较两个对象。

（二）hashCode()

hashCode()是从对象到散列值的映射，对于等价的对象，必须映射到相同的hash值，而不同的对象的映射可以相同，也可以不同，但如果相同会对执行效率有影响。

List和Set是Java中使用十分频繁的两种数据结构，经常要对List和Set中的实例进行比较，这种是只与内部的ADT有关的，但是还有的是对两个List或两个Set进行比较，这种情况因为List和Set是可变的数据类型，内部存储的数据也可能时可变的，容易因为修改而出错。

（四）Mutable数据类型的equals()

1、观察等价性：在不改变状态的情况下，两个mutable对象是否看起来一致
2、行为等价性：调用对象的任何方法都展示出一致的结果
对可变数据来说，往往实现观察等价性，但观察等价性可能导致bug，如前面的HashSet。在JDK中，不同的mutable类实现不同的等价性标准，要根据spec具体判断分析。
在开发ADT时，对于mutble数据类型，实现行为等价性即可，即只有指向相同内存空间的对象才是相等的。所以无需重写，继承Object类的equals()和hashCode()即可。如果要判断观察等价性，最好定义一个新的方法。

（五）equals()和hashCode()不对等

hashCode()是一个对象到散列值的映射，根据hashCode()的原则，equals的对象必须拥有相同的hashCode(),所以hashCode()可以用于对象的初步比较，如果hashCode不等，则两个对象一定不等价。这种比较方式对于一些比较复杂的ADT效率会更高，在需要多次比较时，可以先用hashCode比较，相等后再调用equals(),来降低运行时间。

再前面的示例中，可以看出HashSet,HashMap都是基于hash实现的，而ArrayList的实现使用到hash的情况不多。如果两个对象是equals的但hashCode()不同，对于ArrayList,因为内部实现几乎没有使用hashCode(),所以可以容错，几乎没有影响，但是对于大量使用hash的HashSet和HashMap,两种数据结构的比较都是基于上一段中提到的先比HashCode再比较equals()。例如下面是HashMap的一部分代码，在HashMap的put和HashSet的add操作中都要使用，所以如果hashCode()与equals()不对应，则在equals()正常返回的情况下也不能识别出这是相同的对象，因此在添加时都会被加入set中，从而造成错误。

所以，在实际中，equals()一旦重写，hashCode()必须重写，并进行正确性的测试。或着使用容错更好的ArrayList代替，但这种方法不建议使用，可能因为前期没有暴露错误在后期造成更大的影响。
